---
description: 調査・分析・回答に特化し、コードの変更を一切行わないワークフロー
---

このワークフローは、エージェントがコードベースの調査、設計の相談、ドキュメントの解説、または特定の課題に対する分析のみを行い、**ソースコードやファイルを一切変更しない**ことを保証するための手順です。

## 1. ワークフローの目的

- 既存コードの仕様理解
- 新機能の設計案の作成
- バグの原因特定（修正は行わない）
- ライブラリやフレームワークの使用方法の調査
- リファクタリングの方針検討

## 2. 実行手順

1. **コンテキストの把握**: ユーザーの質問や課題を深く理解します。
2. **コードベースの検索**: `grep_search` や `find_by_name` を使用して、関連するコード箇所を特定します。
3. **内容の精査**: `view_file` や `view_file_outline` を使用して、コードの詳細なロジックを確認します。
4. **外部情報の調査**: 必要に応じて `search_web` や `read_url_content` を使用し、最新のドキュメントや外部の解決策を調査します。
5. **回答の構成**: 得られた情報を整理し、論理的で分かりやすい回答を作成します。

## 3. 厳守事項（禁止事項）

このワークフローが適用されている間、エージェントは以下の行為を**絶対に行いません**。

- **ファイル変更の禁止**: `write_to_file`, `replace_file_content`, `multi_replace_file_content` などのツールを使用して、ワークフローファイル自体を除くプロジェクト内のファイルを変更すること。
- **破壊的コマンドの禁止**: `run_command` を使用して、データの削除、ビルドの実行、コミットなどの「状態を変更する」操作を行うこと（読み取り専用の `ls` や `cat` 等は許可されますが、基本的には読み取りツールを使用してください）。
- **自動実行の制限**: `SafeToAutoRun` を `true` に設定したコマンド実行は控えます。

## 4. 回答の形式

回答は以下の構成で提供します。

- **【概要】**: 調査の結果を簡潔にまとめます。
- **【詳細】**: 関連するファイルパスやコードの引用、外部ドキュメントの参照などを詳しく説明します。
- **【提案/結論】**: 分析に基づく推奨事項や回答を提示します。

## 5. 調査完了後の対応

調査結果に対してユーザーから「修正をお願いします」などの具体的な変更依頼があった場合にのみ、別の適切なワークフローに移行するか、通常のコーディングモードに戻って作業を開始します。
